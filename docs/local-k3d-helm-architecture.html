Smart Coffee - Local k3d + Helm (Simple View)

Minimal architecture for local development and observability.

1) Deploy Flow

Local Machine
Helm + kubectl
→
k3d Cluster
coffee-cluster
Helm installs app + MySQL, and optionally observability stack.
App image is built locally and imported into k3d.
  1. docker build -t smart-coffee:latest ./app creates the app image locally.
  2. k3d image import smart-coffee:latest -c coffee-cluster makes the image available inside the cluster.
  3. helm upgrade --install smart-coffee ./helm/smart-coffee -f values.local.yaml renders templates and applies resources (Deployments, Services, ConfigMap, Secret).
  4. kubectl get pods,svc verifies resources are running and reachable.

2) App + DB Runtime

Service: coffee-api
port 80 → pod 8080
→
Pod: coffee-api
→
Service: mysql
port 3306
→
Pod: mysql
Coffee API reads config from ConfigMap and DB password from Secret.

3) Local Access

Browser
→
kubectl port-forward svc/coffee-api 8080:80
→
API: http://localhost:8080
Browser
→
kubectl port-forward svc/obs-grafana -n monitoring 3000:80
→
Grafana: http://localhost:3000
Helm creates cluster resources; kubectl is used to operate them day-to-day (inspect, logs, port-forward, exec, rollout).
  1. Helm manages desired state from chart templates and values files.
  2. kubectl handles runtime operations like kubectl logs, kubectl describe, and kubectl port-forward.
  3. When config changes, run Helm again to update resources; use kubectl to confirm rollout health.

4) Helm + kubectl Responsibilities

Helm
Package + template + release management
↔
kubectl
Inspect + troubleshoot + access running resources
Think of Helm as "apply many templated Kubernetes objects together" and kubectl as "operate and debug what is currently running".
